# Claude를 잘  
쓰기 위한 기준

Claude를 처음 쓰는 사람부터 팀 단위로 운영하려는 사람까지, 환경 설계부터 보안·의존성·리뷰까지 단계적으로 안내합니다. Claude Code CLI 기준이며 API·SDK에도 공통 적용됩니다. 

각 섹션은 세 층으로 구성됩니다. 필요한 깊이까지만 읽으세요.

왜 — 개념과 이유. 누구나 원칙 — 무엇을 해야 하는가. 팀 리더 실전 — 구체적 예시. 개발자

## 1부 — 에이전트 하네스 : 환경 설계

AGENT HARNESS · ENVIRONMENT DESIGN

## 1-1. 하네스란 무엇인가

왜 이 개념이 왜 필요한가

Claude에게 "버그 고쳐줘"라고만 하면 Claude는 상황을 스스로 해석해서 동작합니다. 이 해석이 의도와 다를 때 엉뚱한 결과가 나와요.

하네스(Harness)는 Claude가 동작하기 전에 **역할, 규칙, 허용 범위를 미리 정의해두는 설정 레이어** 입니다. 안전벨트처럼 Claude의 행동 범위를 잡아주는 역할이에요.

하네스를 잘 설계하면 매번 긴 설명 없이도 Claude가 의도대로 동작하고, 실수나 위험한 행동을 사전에 방지하며, 팀 전체가 일관된 방식으로 Claude를 사용할 수 있습니다.

원칙 무엇을 해야 하는가

  * Claude는 명시된 것만 알고 있다고 가정하세요. 당연한 것도 적어두세요.
  * 역할, 규칙, 금지사항을 분리해서 명시하세요.
  * 하네스는 한 번 만들고 끝이 아니라 쓰면서 계속 다듬는 것입니다.

실전 Claude의 동작 흐름
    
    
    [사용자 요청]
        ↓
    [하네스 레이어]  ← CLAUDE.md, 시스템 프롬프트, 툴 설정
        ↓
    [Claude 실행]
        ↓
    [결과 반환]

## 1-2. CLAUDE.md — 영구 지시서

왜이 개념이 왜 필요한가

매 세션마다 "너는 백엔드 개발자야, Python 써, 타입 힌트 필수야"를 반복하는 건 비효율적이에요. Claude Code는 프로젝트 루트에 `CLAUDE.md` 파일이 있으면 세션 시작 시 자동으로 읽습니다.

한 번 잘 써두면 이후 모든 세션에서 Claude가 같은 기준으로 동작합니다.

원칙무엇을 해야 하는가

  * 역할과 목적을 가장 먼저 명시하세요.
  * 기술 스택과 코드 컨벤션을 구체적으로 적으세요.
  * 금지사항은 명확하게 선언하세요.
  * 너무 길면 오히려 무시됩니다. 핵심만 담으세요.

실전CLAUDE.md 기본 구조

CLAUDE.md
    
    
    # 프로젝트명
    
    ## 역할
    당신은 백엔드 개발자입니다.
    이 프로젝트의 FastAPI 서버를 개발하고 유지보수합니다.
    
    ## 기술 스택
    - 언어: Python 3.11 / 프레임워크: FastAPI / DB: PostgreSQL
    
    ## 코드 규칙
    - 함수명은 snake_case
    - 모든 함수에 docstring 작성 / 타입 힌트 필수
    
    ## 의존성 규칙
    - 새 패키지 추가 전 반드시 사용자 승인 받기
    - requirements.txt 버전 고정 필수
    
    ## 하면 안 되는 것
    - .env 파일 내용 출력 금지
    - 프로덕션 DB 직접 수정 금지
    - 승인 없이 외부 라이브러리 추가 금지
    
    ## 작업 완료 시
    - 변경한 파일 목록 요약
    - 테스트 실행 결과 포함

## 1-3. 시스템 프롬프트 (API / SDK 사용 시)

왜이 개념이 왜 필요한가

Claude Code CLI가 아닌 API나 SDK로 Claude를 직접 호출할 때는 `CLAUDE.md` 대신 시스템 프롬프트로 역할을 정의합니다. 애플리케이션 안에 Claude를 내장할 때 사용하는 방식이에요.

시스템 프롬프트 없이 호출하면 Claude는 일반 대화 모드로 동작합니다. 특정 역할과 규칙이 필요한 서비스라면 반드시 설정해야 해요.

원칙무엇을 해야 하는가

  * 역할 → 목적 → 행동 원칙 → 출력 형식 → 제약 순서로 작성하세요.
  * 출력 형식을 지정하지 않으면 매번 다른 포맷으로 응답합니다.
  * 너무 많은 제약을 걸면 정상적인 작업도 거부할 수 있어요.

실전시스템 프롬프트 기본 구조
    
    
    당신은 [역할]입니다.
    
    ## 목적
    [무엇을 위해 존재하는지]
    
    ## 행동 원칙
    1. 불확실하면 실행 전 반드시 확인
    2. 민감한 정보는 출력하지 않음
    3. 파괴적 작업 전 사용자 동의 필수
    
    ## 출력 형식
    [항상 JSON으로 / 마크다운으로 / 한국어로]
    
    ## 제약
    [하지 말아야 할 것]

## 1-4. 툴 설정 — 허용 범위 결정

왜이 개념이 왜 필요한가

Claude Code에서 Claude는 파일을 읽고 쓰고 터미널 명령어를 실행할 수 있습니다. 이 권한을 무제한으로 열어두면 의도치 않은 파일 삭제나 시스템 명령어 실행이 일어날 수 있어요.

어떤 툴을 허용할지 명시적으로 결정하는 것이 안전한 운영의 기본입니다.

원칙무엇을 해야 하는가

  * 작업에 필요한 툴만 허용하세요 (최소 권한 원칙).
  * 처음에는 좁게 시작하고 필요할 때 넓혀가세요.
  * 툴 범위를 CLAUDE.md에 명시해두세요.

실전툴별 위험도

툴| 설명| 위험도  
---|---|---  
`Read`| 파일 읽기| 낮음  
`Glob`| 파일 탐색| 낮음  
`Write`| 파일 쓰기/수정| 중간  
`WebFetch`| 웹 페이지 조회| 중간  
`WebSearch`| 웹 검색| 중간  
`Bash`| 터미널 명령어 실행| 높음  
  
CLAUDE.md — Bash 범위 명시
    
    
    ## 허용된 Bash 명령어
    - 테스트: pytest, npm test
    - 빌드: make build, npm run build
    - 조회: ls, cat, grep, find
    
    ## 금지된 Bash 명령어
    - rm -rf 류의 삭제 명령
    - 프로덕션 서버 접근 (ssh prod-*)
    - 배포 스크립트 직접 실행 (deploy.sh)

## 1-5. 멀티 세션 구성

왜이 개념이 왜 필요한가

하나의 큰 프로젝트를 혼자 처리하면 Claude의 컨텍스트 한계에 부딪히거나 작업이 뒤섞입니다. 역할을 나눠서 여러 세션으로 운영하면 각 세션이 집중할 범위가 명확해지고 간섭이 줄어요.

원칙무엇을 해야 하는가

  * 역할 경계를 명확히 하세요. 겹치는 영역이 많으면 충돌이 생겨요.
  * 공통 규칙은 루트 CLAUDE.md에, 세션별 규칙은 별도 파일에 분리하세요.
  * 오케스트레이터 세션 하나가 전체 진행 상황을 조율하게 하세요.

실전디렉토리 구조 및 세션 시작
    
    
    프로젝트 루트/
    ├── CLAUDE.md                ← 공통 규칙 (모든 세션이 읽음)
    ├── .claude/
    │   ├── session-backend.md   ← 백엔드 세션 역할 정의
    │   ├── session-frontend.md  ← 프론트엔드 세션 역할 정의
    │   └── session-orch.md      ← 오케스트레이터 역할 정의
    └── .orchestrator/
        ├── status-backend.md    ← 백엔드 진행 상태
        └── status-frontend.md   ← 프론트엔드 진행 상태

### 세션 시작 시
    
    
    .claude/session-backend.md 파일을 읽고 그 역할로 동작해줘.

실전여러 레포/폴더에 접근 권한 주기

Claude Code는 기본적으로 실행한 디렉토리만 알아요. 모든 세션이 여러 레포에 접근해야 한다면 `additionalDirectories`로 접근 권한을 부여해요. **어디에 설정하느냐에 따라 적용 범위가 달라집니다.**

### 설정 위치 3가지
    
    
    // ① 전역 — 모든 세션, 모든 프로젝트에 항상 적용
    // ~/.claude/settings.json
    {
      "permissions": {
        "additionalDirectories": [
          "../bip-frontend",
          "../bip-shared",
          "../bip-pipeline"
        ]
      }
    }
    
    // ② 프로젝트 — 팀 공유 (git 커밋)
    // .claude/settings.json
    {
      "permissions": {
        "additionalDirectories": ["../bip-shared"]
      }
    }
    
    // ③ 로컬 — 개인 경로 (git 제외, .gitignore 등록)
    // .claude/settings.local.json
    {
      "permissions": {
        "additionalDirectories": ["/home/user/workspace/bip-docs"]
      }
    }

### 세션 중 임시 추가 — /add-dir
    
    
    # 실행 중인 세션에 즉시 추가 (권한도 함께 부여)
    /add-dir ../bip-frontend
    
    # 또는 시작 시 플래그로
    claude --add-dir ../bip-frontend --add-dir ../bip-shared

### 각 레포의 CLAUDE.md도 함께 로드하려면
    
    
    // settings.json에 env 추가 (v2.1.20+)
    {
      "env": {
        "CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD": "1"
      },
      "permissions": {
        "additionalDirectories": ["../bip-frontend", "../bip-shared"]
      }
    }

### 멀티 레포 워크스페이스 구조
    
    
    workspace/                       ← 여기서 claude 실행
    ├── CLAUDE.md                    ← 공통 규칙
    ├── CONTEXT.md                   ← 현재 작업 상태 (세션 간 공유)
    ├── .claude/
    │   ├── settings.json            ← additionalDirectories 설정
    │   └── settings.local.json      ← 개인 경로 (git 제외)
    ├── bip-frontend/
    │   └── CLAUDE.md
    ├── bip-pipeline/
    │   └── CLAUDE.md
    └── bip-shared/
        └── CLAUDE.md

### CONTEXT.md — 세션 간 상태 공유 파일
    
    
    ## 현재 작업 (2026-04-12)
    태스크: NL2SQL 엔드포인트 + Oracle 연동
    
    ## 완료
    - bip-pipeline: FastAPI 라우터 기본 구조
    - bip-shared: OracleConnector 클래스
    
    ## 진행 중
    - bip-pipeline: nl2sql.py SQL 검증 로직 (미완)
    
    ## 중요 결정사항
    - ROWNUM 대신 FETCH FIRST 사용

### 세션 시작 프롬프트 (멀티 레포 환경)
    
    
    CONTEXT.md와 .claude/session-backend.md를 먼저 읽고,
    현재 상태를 파악한 후 진행 중인 작업부터 이어서 시작해줘.

**⚡ 탐색은 넓게, 수정은 좁게**  
additionalDirectories로 여러 레포를 읽어오는 건 자유롭게 해도 돼요. 단, 한 세션에서 여러 레포를 동시에 수정하면 컨텍스트가 빠르게 차고 혼란이 생겨요. 실제 수정 작업은 해당 레포 세션에 집중하세요. 

왜이 개념이 왜 필요한가

실제 프로젝트는 레포가 여러 개인 경우가 많아요. 프론트엔드, 백엔드, 공유 라이브러리, 파이프라인이 각자 다른 폴더에 있는데, 모든 세션이 관련 레포에 **동시에 접근** 해야 하는 상황이 생겨요.

Claude Code는 기본적으로 실행한 디렉토리만 알고 있어요. 다른 레포에 접근하려면 **추가 디렉토리 설정** 이 필요해요. 이 설정을 어디에 넣느냐에 따라 효과 범위가 달라져요.

원칙무엇을 해야 하는가

  * **모든 세션에 항상 필요한 레포** → 전역 설정 (`~/.claude/settings.json`)
  * **팀 전체가 공유해야 하는 레포** → 프로젝트 설정 (`.claude/settings.json`, git 커밋)
  * **내 로컬 경로만 다른 경우** → 로컬 설정 (`.claude/settings.local.json`, git 제외)
  * 한 세션에 모든 레포를 몰아넣지 마세요. 컨텍스트가 빠르게 차고 혼란이 생겨요.

실전설정 위치별 방법

### ① 전역 설정 — 모든 세션에 항상 적용
    
    
    // ~/.claude/settings.json
    {
      "permissions": {
        "additionalDirectories": [
          "../bip-frontend",    ← 항상 접근 허용할 레포들
          "../bip-shared",
          "../bip-pipeline"
        ]
      }
    }

### ② 프로젝트 설정 — 팀 공유 (git 커밋)
    
    
    // .claude/settings.json  ← git에 커밋, 팀 전체 적용
    {
      "permissions": {
        "additionalDirectories": ["../bip-shared"],
        "deny": ["Read(.env)"]
      }
    }

### ③ 로컬 설정 — 개인 경로 (git 제외)
    
    
    // .claude/settings.local.json  ← .gitignore에 추가, 개인용
    {
      "permissions": {
        "additionalDirectories": ["/home/user/workspace/bip-docs"]
      }
    }

### ④ 세션 중 임시 추가 — /add-dir 슬래시 커맨드
    
    
    # 실행 중인 세션에 즉시 추가, 권한도 함께 부여됨
    /add-dir ../bip-frontend
    /add-dir ../bip-shared
    
    # 또는 시작 시 플래그로
    claude --add-dir ../bip-frontend --add-dir ../bip-shared

### ⑤ 추가 디렉토리의 CLAUDE.md도 함께 로드
    
    
    // settings.json에 env 추가 → 각 레포의 CLAUDE.md도 자동 로드
    {
      "env": {
        "CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD": "1"
      },
      "permissions": {
        "additionalDirectories": ["../bip-frontend", "../bip-shared"]
      }
    }

## 1-6. SKILL.md — 작업 노하우 모듈

왜이 개념이 왜 필요한가

CLAUDE.md가 **"Claude가 어떤 존재인가"** 를 정의한다면, SKILL.md는 **"Claude가 특정 작업을 어떻게 잘 해내는가"** 를 정의합니다.

숙련된 선임이 후임에게 남기는 작업 매뉴얼과 같아요. "이 작업은 이 라이브러리 써라, 이 순서로 해라, 이런 실수 조심해라"를 문서로 남겨두면 매번 같은 설명을 반복하지 않아도 돼요.

Claude Code는 작업 요청이 들어오면 관련 Skill 파일이 있는지 자동으로 확인해요. 있으면 읽고 그 방식대로 작업하고, 없으면 자기 판단으로 진행합니다.

원칙무엇을 해야 하는가

  * 같은 유형의 작업을 3번 이상 반복하게 될 때 Skill로 만드세요.
  * 실수가 반복되는 패턴, 매번 긴 설명이 필요한 작업이 좋은 후보예요.
  * 팀이 Git으로 공유하면 Skill도 자동으로 함께 공유돼요.
  * 모든 작업을 Skill로 만들 필요는 없어요. CLAUDE.md 수준으로 해결되는 건 굳이 만들지 마세요.

실전구조 및 작성 예시

### 디렉토리 구조
    
    
    프로젝트 루트/
    └── .claude/
        └── skills/
            ├── nl2sql/

            ├── demand-forecast/

            └── data-pipeline/
                └── SKILL.md     ← 파이프라인 작업 방법

    # NL2SQL 스킬
    
    ## 이 스킬을 쓰는 상황
    - 사용자가 자연어로 데이터 조회를 요청할 때
    - SQL 초안 생성 후 검증이 필요할 때
    
    ## 작업 순서

    - SQL과 함께 한국어 설명 포함

## 1-7. MCP — 외부 시스템 연결

왜이 개념이 왜 필요한가

Claude는 기본적으로 대화 안에서만 동작해요. 실제 DB에서 데이터를 가져오거나, Slack에 메시지를 보내거나, GitHub에서 코드를 읽으려면 외부 시스템과 연결이 필요합니다.

MCP(Model Context Protocol)는 **Claude가 외부 시스템과 대화할 수 있게 해주는 표준 연결 방식** 이에요. USB 포트처럼 규격이 정해져 있어서, 그 규격에 맞게 만든 서버라면 어떤 시스템이든 동일한 방식으로 연결할 수 있어요.

SKILL.md가 "어떻게 할지 아는 것"이라면, MCP는 "실제로 연결하는 것"이에요. 둘은 역할이 달라서 함께 씁니다.

원칙무엇을 해야 하는가

  * 외부 시스템 접근이 필요할 때만 MCP를 추가하세요.
  * 각 MCP 서버의 권한 범위를 명확히 파악하세요.
  * 사내 환경에서는 허용된 MCP 서버 목록을 팀 내 합의하세요.
  * 민감한 시스템(프로덕션 DB 등)은 MCP 연결 자체를 제한하세요.

실전세 개념 비교 및 연결 방식

### 세 개념 한눈에 비교

구분| 역할| 비유  
---|---|---  
`CLAUDE.md`| 역할과 규칙 정의| 명함 + 사규  
`SKILL.md`| 특정 작업 노하우| 업무 매뉴얼  
`MCP`| 외부 시스템 연결| 전화기 + 사내 시스템  
  
### 실제 동작 흐름 예시 — NL2SQL
    
    
    사용자: "지난 분기 아이폰 판매량 알려줘"
        ↓
    CLAUDE.md 읽음  → "나는 데이터 분석가. Oracle DB 담당."
        ↓
    SKILL.md 읽음   → "NL2SQL은 메타데이터 먼저 확인하고,
                       Oracle 문법 써서, EXPLAIN 검증해라"
        ↓
    MCP 사용        → Oracle DB에 실제 접속해서 쿼리 실행
                       결과 가져옴
        ↓
    사용자에게 답변

### MCP 연결 가능한 시스템 예시
    
    
    Claude
      ├── MCP → Oracle DB        (데이터 조회)
      ├── MCP → Slack            (메시지 전송)
      ├── MCP → GitHub           (코드 읽기/쓰기)
      ├── MCP → Google Drive     (문서 접근)
      └── MCP → 사내 API 서버    (커스텀 연결)

### 언제 뭘 쓰는가

상황| 필요한 것  
---|---  
팀 코딩 컨벤션 가르치기| `CLAUDE.md`  
수요 예측 방법론 가르치기| `SKILL.md`  
Oracle DB에서 실제 데이터 가져오기| `MCP`  
Slack에 메시지 보내기| `MCP`  
특정 작업을 일관되게 잘 하게 하기| `SKILL.md`  
외부 시스템과 실제로 연동하기| `MCP`

## 2-1. 기본 원칙

왜이 개념이 왜 필요한가

Claude는 지시가 명확할수록 좋은 결과를 냅니다. 사람에게 업무를 맡길 때처럼 역할, 맥락, 기대 결과를 구체적으로 전달할수록 원하는 답이 나와요.

"버그 고쳐줘"와 "Python 시니어 개발자로서 이 코드의 버그를 찾아 수정하고 원인을 설명해줘"의 결과물은 전혀 다릅니다.

원칙무엇을 해야 하는가

  * 역할을 먼저 부여하세요.
  * 배경과 맥락을 충분히 제공하세요.
  * 출력 형식을 명시하세요.
  * 예시를 보여주면 훨씬 정확해집니다 (Few-shot).

실전4가지 기본 패턴

### ① 역할 부여
    
    
    # 나쁜 예
    버그 고쳐줘.
    
    # 좋은 예
    당신은 Python 시니어 개발자입니다.
    아래 코드의 버그를 찾아 수정하고
    왜 버그가 발생했는지 설명해주세요.

### ② 맥락 제공
    
    
    # 나쁜 예
    이 함수 최적화해줘.
    
    # 좋은 예
    이 함수는 하루 10만 건 호출됩니다.
    현재 평균 응답시간 200ms인데 100ms 이하로 줄이고 싶습니다.
    메모리 사용량은 현재 수준 유지해주세요.

### ③ 출력 형식 지정
    
    
    결과를 아래 형식의 JSON으로만 출력해주세요:
    {
      "status": "success" | "fail",
      "changes": ["변경사항1", "변경사항2"],
      "reason": "이유"
    }

### ④ 예시 제공 (Few-shot)
    
    
    아래 형식으로 함수명을 변환해주세요.
    
    예시:
    - get_user_data → getUserData
    - send_email_notification → sendEmailNotification
    
    변환할 목록:
    - fetch_product_list
    - update_order_status

## 2-2. 단계적 사고 유도

왜이 개념이 왜 필요한가

복잡한 문제를 Claude에게 한 번에 던지면 빠르게 결론을 내리려다 중요한 부분을 건너뛰는 경향이 있어요. 단계를 나눠서 생각하게 하면 훨씬 꼼꼼한 결과가 나옵니다.

원칙무엇을 해야 하는가

  * 복잡한 작업일수록 먼저 계획을 세우게 하세요.
  * 단계별로 확인하면서 진행하세요.
  * 결론보다 과정을 먼저 검토하는 것이 오류를 줄입니다.

실전단계 분리 예시
    
    
    # 바로 결론 요청 (복잡한 문제에서 비효율적)

    # 단계적 사고 유도
    이 시스템 설계를 검토해줘.
    1단계: 현재 설계 구조 요약
    2단계: 잠재적 문제점 나열
    3단계: 각 문제에 대한 해결안
    4단계: 최종 권고사항
    순서대로 진행해줘.

## 3-1. 민감 정보 관리

왜이 개념이 왜 필요한가

Claude는 대화 내용을 기반으로 동작합니다. API 키, 패스워드, 개인정보가 포함된 파일을 Claude에게 읽히면 의도치 않게 출력되거나 로그에 남을 수 있어요.

사람과 대화할 때도 비밀번호를 말하지 않듯, Claude에게도 민감 정보를 불필요하게 노출하지 않는 습관이 중요합니다.

원칙무엇을 해야 하는가

## 8부 — 컨텍스트 엔지니어링

## 8-1. 컨텍스트란 무엇인가

왜이 개념이 왜 필요한가

프롬프트 엔지니어링이 "Claude에게 어떻게 말을 거는가"라면, 컨텍스트 엔지니어링은 "Claude의 작업 창에 무엇을 담을 것인가"입니다.

Claude는 한 번에 처리할 수 있는 텍스트 양(컨텍스트 창)에 한계가 있어요. 이 공간에 무엇을 넣고 무엇을 빼는지가 결과 품질과 비용 모두에 직결됩니다. 관련 없는 정보가 많을수록 Claude의 집중도가 떨어지고 토큰 비용이 올라가요.

쉽게 말해 **컨텍스트 창은 책상 위의 공간** 이에요. 필요한 것만 올려두면 작업이 빠르고 정확하지만, 온갖 것들이 쌓여 있으면 원하는 걸 찾기도 어렵고 공간도 부족해요.

원칙무엇을 해야 하는가

  * 지금 작업에 필요한 정보만 컨텍스트에 포함하세요.
  * 파일 전체보다 관련 함수나 섹션만 넘기세요.
  * 이전 대화가 길어지면 주기적으로 요약해서 압축하세요.
  * CLAUDE.md가 잘 작성되어 있으면 반복 설명이 줄어들어요.

실전컨텍스트 창 구성 요소
    
    
    Claude의 컨텍스트 창
    ├── 시스템 프롬프트 / CLAUDE.md     ← 고정 영역 (매번 포함)
    ├── SKILL.md (해당 작업 시)          ← 작업별 로드
    ├── 이전 대화 내역                   ← 길어질수록 위험
    ├── 현재 파일/코드 내용              ← 필요한 부분만
    └── 사용자 요청                      ← 현재 작업

### 컨텍스트 효율 비교

방식| 토큰 사용| 결과 품질  
---|---|---  
파일 전체를 통째로 넘김| 높음| 집중도 낮음  
관련 함수/섹션만 발췌| 낮음| 집중도 높음  
이전 대화 전체 유지| 높음| 초반 내용 희석  
중간 요약 후 계속| 낮음| 핵심 유지

## 9부 — 비용 관리

COST MANAGEMENT

## 9-1. 비용이 쌓이는 구조

왜이 개념이 왜 필요한가

Claude API는 입력 토큰 + 출력 토큰 기준으로 과금돼요. 생각보다 빠르게 쌓이는 경우가 많은데, 주로 이 세 가지 때문이에요.

**반복 컨텍스트** — 매 세션마다 같은 배경 설명을 긴 프롬프트로 넣는 것. CLAUDE.md로 해결 가능해요.

**불필요하게 긴 입력** — 파일 전체를 넘기거나, 관련 없는 정보까지 포함하는 것. 컨텍스트 설계로 해결돼요.

**모델 오버스펙** — 간단한 작업에 가장 비싼 모델을 쓰는 것. 작업별 모델 선택으로 해결돼요.

실전모델별 비용 구조 (2026 기준)

모델| 상대적 비용| 적합한 작업  
---|---|---  
Claude Opus 4.6| 높음| 복잡한 추론, 아키텍처 설계, 어려운 버그  
Claude Sonnet 4.6| 중간| 일반 코딩, 코드 리뷰, 문서 작성  
Claude Haiku 4.5| 낮음| 단순 반복 작업, 포맷 변환, 요약  
  
## 9-2. 모델 선택 전략

왜이 개념이 왜 필요한가

모든 작업에 가장 비싼 모델을 쓸 필요가 없어요. 단순 포맷 변환, 요약, 반복 작업은 저렴한 모델로도 충분히 좋은 결과가 나와요. 모델을 전략적으로 선택하면 품질은 유지하면서 비용을 크게 줄일 수 있어요.

원칙무엇을 해야 하는가

  * 복잡도에 맞는 모델을 선택하세요. 항상 최고 모델을 쓰지 마세요.
  * 반복 작업과 단순 작업은 Haiku로 먼저 시도하세요.
  * 결과가 불만족스러울 때 더 강한 모델로 올리세요.
  * 오케스트레이터는 강한 모델, 서브에이전트는 저렴한 모델을 쓰세요.

실전작업별 모델 선택 가이드

작업 유형| 권장 모델| 이유  
---|---|---  
시스템 아키텍처 설계| Opus 4.6| 복잡한 트레이드오프 판단 필요  
어려운 버그 디버깅| Opus 4.6| 깊은 추론 능력 필요  
일반 기능 구현| Sonnet 4.6| 품질과 비용 균형  
코드 리뷰| Sonnet 4.6| 패턴 인식 충분  
문서/주석 작성| Haiku 4.5| 언어 작업, 추론 불필요  
포맷 변환, 요약| Haiku 4.5| 단순 변환 작업  
반복 배치 작업| Haiku 4.5| 대량 처리, 비용 중요  
  
## 9-3. 캐싱과 배치 처리

왜이 개념이 왜 필요한가

같은 시스템 프롬프트나 컨텍스트를 반복해서 넘기는 경우, Anthropic의 프롬프트 캐싱을 활용하면 입력 토큰 비용을 크게 줄일 수 있어요. 또한 긴급하지 않은 대량 작업은 배치 API로 처리하면 비용이 절반으로 줄어요.

실전비용 절감 방법 3가지

### ① 프롬프트 캐싱 (API 사용 시)
    
    
    # 긴 시스템 프롬프트나 문서는 캐싱 활용
    # 첫 호출: 전체 토큰 비용
    # 이후 호출: 캐시 히트 → 입력 토큰 90% 절감
    
    messages=[
        {
            "role": "user",
            "content": [
                {
                    "type": "text",
                    "text": long_document,
                    "cache_control": {"type": "ephemeral"}
                    # ↑ 이 블록을 캐싱
                },
                {"type": "text", "text": user_question}
            ]
        }
    ]

### ② 배치 API (대량 처리 시)
    
    
    # 긴급하지 않은 대량 작업 → Message Batches API
    # 일반 API 대비 50% 비용 절감
    # 24시간 내 처리 (실시간 불필요한 작업에 적합)
    
    # 적합한 작업: 대량 문서 요약, 코드 분석, 데이터 변환
    # 부적합한 작업: 대화형 작업, 즉시 응답 필요한 작업

### ③ CLAUDE.md로 반복 컨텍스트 제거
    
    
    # 매 세션마다 반복하던 설명 → CLAUDE.md에 한 번만
    # 프롬프트 길이 감소 → 토큰 절감
    
    # 절감 예시:
    # 반복 설명 없이: 평균 500 토큰/세션
    # CLAUDE.md 활용: 평균 100 토큰/세션 (80% 절감)
